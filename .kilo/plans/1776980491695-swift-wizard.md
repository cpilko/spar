# Plan: Replace DuckDuckGo Search with Better Alternative

## Current Implementation

The current search in `spar.py` uses a custom `FunctionTool` that:
- Makes HTTP GET requests to `https://html.duckduckgo.com/html/` via `httpx`
- Returns raw HTML (truncated to 50k chars)
- Model must parse raw HTML to extract information

**Problems:**
1. DuckDuckGo's HTML interface is unreliable / rate-limited
2. Raw HTML parsing is error-prone and token-inefficient
3. No structured results (URLs, titles, snippets)
4. Client-side solution that may get blocked

---

## CRITICAL FINDING: OpenRouter Server Tools NOT Compatible with Current Setup

**After investigating the agents SDK source code**, I found that:

> `Converter.tool_to_openai()` in the agents SDK **only supports `FunctionTool`** types and explicitly raises an error for server/hosted tools:
> ```
> "Hosted tools are not supported with the ChatCompletions API"
> ```

This means the `OpenAIChatCompletionsModel` used by spar.py **cannot use OpenRouter server-side tools** (`openrouter:web_search`, `openrouter:web_fetch`). The Chat Completions API path used by the SDK doesn't support mixing server tools with custom function tools.

---

## Option 1: Direct Exa API (RECOMMENDED)

### How it works
Replace the httpx call to DuckDuckGo with the `exa-py` SDK. Your code makes the API call to Exa, processes the structured results, and returns them to the model via the existing `FunctionTool` mechanism.

### Pros
- **Your existing Exa credits**: Use credits you already have
- **Better than DuckDuckGo**: Exa is designed for AI search with semantic understanding
- **Structured results**: Returns URLs, titles, and highlights (relevant excerpts)
- **Token-efficient**: Highlights extract only relevant content (not full HTML)
- **Categories**: Can search specific indexes (companies, people, research papers, news)
- **Highlighting**: Only returns the most relevant parts of each result

### Cons
- **Client-side**: Same pattern as current implementation (your code calls Exa)
- **Custom tool wrapper needed**: Still uses `FunctionTool`, but the implementation changes

### Implementation
```python
from exa_py import Exa

exa = Exa(api_key=os.environ.get("EXA_API_KEY"))

# In web_fetch_tool_impl:
results = exa.search(
    query,
    type="auto",  # or "instant" for faster, "deep" for complex research
    contents={"highlights": {"max_characters": 4000}}
)
# Format results as string with URL, title, and highlights
formatted = "\n\n".join([
    f"Title: {r.title}\nURL: {r.url}\nSummary: {r.highlights}"
    for r in results.results
])
```
